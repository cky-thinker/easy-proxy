## 目标
- 为列出的服务类的“新增”和“更新”方法补齐并统一业务校验。
- 复用相同校验逻辑：将分散在方法内的零散校验收敛到可复用的私有方法（或轻量工具方法），减少重复与遗漏。
- 保持对外 API 与返回结构不变，仅提升健壮性与一致性。

## 复用策略
- 每个 Service 内新增两个私有方法：`validateForCreate(entity)` 与 `validateForUpdate(existing, patch)`，在对应新增/更新入口调用。
- 已存在的细粒度校验（如端口范围、host:port 格式、唯一性等）继续复用；缺失处补齐调用路径。
- 可选：新增一个轻量通用校验工具类 `Validators`（仅包含通用断言：非空、非负、端口范围、host:port 格式）。若仅被单一服务使用，则优先保留在服务私有方法中，避免过度抽象。

## 逐文件修改

### 1) ProxyClientRuleService.java
- 现状：新增已调用多处私有校验；更新方法内存在局部重复与分散校验（部分未复用已有方法）。
- 变更：
  - 新增：`private void validateForCreate(ProxyClientRule rule)` 聚合以下调用：
    - `validateProxyClientExists(rule.getProxyClientId())`
    - `validateRuleNameUnique(rule.getProxyClientId(), rule.getName(), null)`
    - `validateServerPortRangeAndUnique(rule.getServerPort(), null)`
    - `validateClientAddress(rule.getClientAddress())`
    - `validateLimits(rule.getLimitConn(), rule.getLimitRate())`
  - 新增：`private void validateForUpdate(ProxyClientRule existing, ProxyClientRule patch)`：
    - 禁止变更 `proxyClientId`（保持原逻辑）
    - 若 `name` 变更：`validateRuleNameUnique(existing.getProxyClientId(), patch.getName(), patch.getId())`
    - 若 `serverPort` 变更：`validateServerPortRangeAndUnique(patch.getServerPort(), patch.getId())`
    - 若 `clientAddress` 变更：`validateClientAddress(patch.getClientAddress())`
    - 若 `limitConn/limitRate` 提供：`validateLimits(...)`
  - `addProxyClientRule` 与 `updateProxyClientRule` 入口分别调用 `validateForCreate/validateForUpdate`，并将更新字段的复制与审计字段处理保留原样。
- 参考位置：
  - 新增逻辑汇总：`.../ProxyClientRuleService.java:129-141`
  - 更新方法重构：`.../ProxyClientRuleService.java:146-213`

### 2) ProxyClientService.java
- 现状：已具备 `validateNameUnique/validateTokenFormat/validateTokenUnique`；新增/更新分散调用，逻辑完整但可收敛。
- 变更：
  - 新增：`private void validateForCreate(ProxyClient client)`：依次调用名称唯一、Token 格式、Token 唯一；并不改动原有默认值设置。
  - 新增：`private void validateForUpdate(ProxyClient existing, ProxyClient patch)`：
    - `name` 变更：`validateNameUnique(patch.getName(), patch.getId())`
    - `token` 变更：`validateTokenFormat(...)` + `validateTokenUnique(..., patch.getId())`
  - `addProxyClient`/`updateProxyClient` 中改为调用上述聚合方法，删除重复分支判断。
- 参考位置：
  - 新增方法：`.../ProxyClientService.java:87-98`
  - 更新方法：`.../ProxyClientService.java:103-132`

### 3) UserService.java
- 现状：有 `validateUniqueUserFields/validatePassword/validateRole`；新增已调用，更新未校验 `role` 合法值。
- 变更：
  - 新增：`private void validateForCreate(SysUser user)`：
    - 校验：`username` 非空、`validatePassword(user.getPassword(), true)`、`validateRole(user.getRole())`、`validateUniqueUserFields(user.getUsername(), user.getMobile(), user.getEmail(), null)`
  - 新增：`private void validateForUpdate(SysUser existing, SysUser patch)`：
    - 唯一性：对变更的 `username/mobile/email` 计算差异后调用 `validateUniqueUserFields(..., patch.getId())`
    - 角色：若 `role` 提供，调用 `validateRole(patch.getRole())`
    - 保持“此接口不允许改密码”的约束
  - `createUser`/`updateUser` 调整为先调 `validateForCreate/validateForUpdate` 再落库。
- 参考位置：
  - 新增方法：`.../UserService.java:219-229`
  - 更新方法：`.../UserService.java:231-257`（在设置字段前加入 `validateForUpdate` 与 `validateRole` 调用）

### 4) SysLogService.java
- 现状：仅设置 `createTime`，缺少基本必填校验；无更新方法。
- 变更：

    - `logContent` 非空
    - 若项目存在固定枚举/常量集（如 `INFO/WARN/ERROR`），则校验取值范围（后续按实体/常量类落地）。
  - `addSysLog` 调用 `validateForCreate` 后再插入。
- 参考位置：`.../SysLogService.java:58-62`

## 可选的通用工具类（若需要）
- 新增 `com.cky.proxy.server.util.Validators`：
  - `requireNotBlank(String, String message)`
  - `requireNonNegative(Integer/Long, String fieldName)`
  - `requirePortInRange(int)`（1-65535）
  - `requireHostPort(String)`（格式 `host:port` 且端口范围合法）
- 在仅一个服务使用的场景，优先保留服务内私有校验，避免过度抽象。

## 兼容性与回滚
- 不更改外部 API 签名与响应结构；仅在违规输入时更早、统一抛出 `RuntimeException` 与既有文案。
- 如触发新校验导致少量历史脏数据更新失败，可在控制台日志中定位；回滚仅需恢复到改动前版本。

## 验证方案
- 控制层已使用 `ValidateUtil` 的 Bean Validation 分组；本次聚合确保服务层业务规则完整执行。
- 为以下关键路径补充/更新用例（或 Postman/接口联调）：
  - 用户：新增重复用户名/手机号/邮箱；更新变更角色非法值；更新重复邮箱
  - 客户端：新增非法 Token、重复 Token；更新变更为重复名称
  - 规则：新增冲突端口/非法地址/负数限额；更新跨客户端重名、端口冲突
  - 系统日志：新增缺少 `logType`/`logContent`
- 通过日志与返回信息核验所有异常路径提示一致、可读。
